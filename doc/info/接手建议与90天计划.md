# Codeman 接手建议与 90 天计划

更新日期：2026-03-26

## 1. 先说结论

这个项目不适合一上来就大改架构，也不适合马上扩大功能面。

更合理的接手顺序是：

1. 先建立认知和数据
2. 再确认产品主战场
3. 最后再做结构性重构和商业化试探

## 2. 接手第一周要完成的事

### 2.1 建立“项目真相板”

你需要先确认的不是代码细节，而是经营和产品真相：

- 当前谁在用
- 最常用的是哪些功能
- 用户最离不开的是移动端、自治恢复、还是多会话
- 线上/实际使用中的主要故障来自哪里
- 发版节奏是否稳定

如果目前没有线上数据，至少先补这些基础记录：

- 每周活跃用户数
- 日均 session 数
- 每个 session 平均时长
- respawn 启用率
- mobile 访问占比
- tunnel 启用率
- 子代理面板使用率
- 崩溃/重连/恢复失败次数

### 2.2 跑通核心路径

接手者必须亲自跑通下面这些流程，且最好录屏留档：

- 本地启动 Web UI
- 创建 session
- 手机访问
- QR 登录
- tunnel 暴露
- respawn 开启与恢复
- 子代理展示
- 会话恢复

理由很简单：这个项目的价值高度依赖端到端体验，单看代码很难真正理解。

### 2.3 做一张系统地图

建议自己补一张内部图，至少画清这几层：

- UI 层
- API/SSE/WS 层
- session/respawn/orchestrator 控制层
- tmux/PTY/外部 CLI 执行层
- auth/tunnel/push 外围层

接手前两周，任何重要改动都应该先映射到这张图上，避免只看到局部。

## 3. 前 30 天目标

### 3.1 产品目标

只做一件事：确认项目的主打价值到底是什么。

我建议优先验证下面四个价值点的真实强弱：

- `Mobile-first agent control`
- `Multi-session dashboard`
- `Respawn autonomous work`
- `Subagent visualization`

不要默认四个都 equally important。很可能真正驱动留存的只有其中一到两个。

### 3.2 研发目标

前 30 天研发上不要追求“重写”，而是做“可控接手”：

- 给关键模块补阅读文档和边界说明
- 梳理核心事件流
- 明确哪些功能最依赖外部 CLI 输出格式
- 建一个“高风险改动清单”
- 建一个“稳定性 issue 分类表”

建议重点盯住这几个核心文件：

- `src/respawn-controller.ts`
- `src/web/server.ts`
- `src/session.ts`
- `src/web/public/app.js`

### 3.3 运营目标

如果你能接触到用户，前 30 天至少做 5 到 10 个深访，重点不是问“喜不喜欢”，而是问：

- 你为什么不用原生 CLI，而要用 Codeman
- 你最常在什么场景打开它
- 哪个功能最不可替代
- 哪些地方让你不放心开一整夜
- 如果要付费，你愿意为什么付费

## 4. 30 到 60 天目标

### 4.1 产品上做收敛，不做发散

这阶段建议开始收敛叙事和信息架构：

- 统一术语
- 收敛首页主卖点
- 简化初次使用路径
- 把高级功能分层，不要全部同时暴露

特别建议处理这些概念的一致性：

- session
- case
- respawn
- Ralph
- subagent
- team

如果术语不收敛，新用户和新同事都会持续迷路。

### 4.2 研发上做“小刀快切”的结构优化

这一阶段适合做的，不是大规模重构，而是针对性降风险：

- 继续拆 `server.ts` / `respawn-controller.ts` / `session.ts` 的职责
- 明确 watcher、parser、orchestrator 的输入输出契约
- 给关键流程补端到端回归用例
- 建立更清晰的故障分类和日志标识

目标是：

> 把“只有老作者能安全改”的区域，逐步变成“新负责人也能安全改”。

### 4.3 商业化上做轻验证

不要急着正式收费，但可以开始验证付费意愿：

- 开一个 waitlist 或专业版意向表单
- 放出 2 到 3 个明显偏高级用户的 feature concept
- 观察哪类用户愿意留下联系方式或愿意付费试用

## 5. 60 到 90 天目标

### 5.1 做一次明确的产品选择

到 90 天时，应该做出一个明确选择，而不是继续模糊推进：

- 选择 A：深耕个人重度用户控制台
- 选择 B：往小团队协作工具延展
- 选择 C：做自托管 + 托管双形态

三条路都能走，但不能同时均匀发力。

### 5.2 如果继续做个人重度用户

建议重点投入：

- 体验极致化
- 移动端完善
- 稳定性和恢复能力
- 更强运行可视化
- 更低接入门槛

### 5.3 如果准备做团队版

优先级应转向：

- 权限与审计
- 共享视图
- 组织级配置
- 成本追踪
- 角色分工

这将是一次产品边界变化，不只是多加几个按钮。

## 6. 我建议你立刻建立的三个清单

### 6.1 不可轻动区

把以下模块列为高风险区，所有改动都要经过额外验证：

- session 生命周期
- respawn 状态机
- terminal batching / anti-flicker
- mobile keyboard / viewport 适配
- auth / QR / tunnel

### 6.2 真正的北极星指标候选

建议从下面几项里选 1 到 2 个做北极星指标候选：

- 周活跃重度用户数
- 日均活跃 session 数
- 单用户每周平均 agent 运行时长
- respawn 启用后的成功连续运行时长
- mobile 访问用户留存

### 6.3 接下来暂时不做的事

为了防止接手后分散火力，我建议先明确“不做”：

- 不急着大规模重写前端框架
- 不急着扩成通用 AI 平台
- 不急着做复杂企业后台
- 不急着增加太多新术语和新概念

## 7. 一个务实的 90 天执行版本

### 第 1 到 2 周

- 跑通全链路
- 建系统地图
- 建数据面板
- 梳理术语表

### 第 3 到 4 周

- 做用户访谈
- 明确主价值排序
- 输出接手版产品路线图

### 第 5 到 8 周

- 做一轮核心复杂度降风险
- 补高价值回归测试
- 简化 onboarding 和产品叙事

### 第 9 到 12 周

- 验证一个商业化方向
- 确定项目未来 6 个月主线
- 对外重新组织 README / 官网 / 演示素材

## 8. 最后建议

这个项目最值得保住的不是“功能数量”，而是它已经形成的那种使用体验和工作方式。

所以接手时，优先级应该是：

- 先保住体验优势
- 再降低维护复杂度
- 最后才是扩功能和扩市场

如果顺序反过来，很容易出现“看起来做了很多，但项目灵魂没了”的情况。
